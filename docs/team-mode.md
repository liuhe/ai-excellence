# 团队模式

> 本工程对 [困难二·关注点管理](three-challenges.md#二关注点管理--角色分工问题) 的当前答案。替代已废弃的 `aie-task-*` 工作流。

## 定位

不是流水线，是组织——用户只跟项目经理（PM）对话，PM 按系统设计师写在 `design.md` 里的指令协调一支 AI 团队。

**四条核心约束**：

1. 流程决策权归技术决策者（系统设计师），不归 PM
2. 决策落 `design.md`，PM 只是"文档执行机"：不解释、不判断、不优化
3. 每个角色有显式反清单——越界即拒答
4. 暂停区分"待确认" vs "卡住"，从机制上防止 AI 自我糊弄

## 角色编制（7 个）

| 角色 | 介入时机 | 只关注 | 不许碰 |
|------|---------|--------|--------|
| **PM 项目经理** | 全程 | 用户对接 / 进度汇报 / 按文档调度 / 写 log | 技术判断 / 优化文档指令 / 自作主张 |
| **产品经理** | PM 判断需求层不清；设计师标记需要澄清 | 用户价值 / 验收口径 / 需求边界 | 技术方案 / 实现细节 |
| **系统设计师** | PM 判断非纯粹改字 | 边界 / 接口契约 / 不变量 / 数据所有权 / 失败模式 / 流程决策 | 具体代码 / 变量命名 / 库选型细节 |
| **设计评审员** | 设计师在 `design.md` 勾选"需要评审" | 挑刺 / 找漏洞 / 标注分级（blocker / major / minor） | 出方案 / 改写设计 |
| **测试用例设计师** | 设计师在 `design.md` 勾选"需要测试用例先设计" | 该测什么 / 覆盖矩阵 / 验收口径 / 失败模式枚举 | 测试代码怎么写 / 实际执行 |
| **开发工程师** | 设计完成且执行到实现步骤 | 实现 / 局部正确性 | 重新设计 / 改需求 |
| **测试工程师** | 开发完成 | 用例实际执行结果 / 回归 / 现场证据 | 重新设计用例（按拿到的清单跑） |

运维 / 运营暂不实现，后续视需要再加。

## 角色 system prompt 的硬约束模板

每个角色 system prompt **第一段是禁令，不是描述**。以系统设计师为例：

```
你是系统设计师。

你只能谈：边界 / 接口契约 / 不变量 / 数据所有权 / 失败模式
你不能谈：具体代码实现 / 变量命名 / 错误处理写法 / 库选型细节

被问到越界问题时必须显式拒答，并指明该找谁：
- "怎么写这段代码" → "这是开发工程师的关注点，请 PM 转交"
- "用 X 还是 Y 库"   → "这是开发工程师的关注点"
- "这个需求值不值得做" → "这是产品经理的关注点"
```

7 个角色都按这个套路写。"该想什么"是软约束，靠提醒；"不许想什么"是硬约束，靠拒答触发 PM 转交——拒答本身是 PM 的调度信号。

## 产品经理 / 系统设计师的输出语言：DCDDP

这两个角色的产出**必须按 DCDDP v6.2 系统建模语言**组织，不写散文：

- **产品经理**：业务视图（参与方 / 业务用例）+ 系统用例（含验收口径）
- **系统设计师**：4 视图分层（业务 / 领域模型 / 系统逻辑 / 必要时部署）+ 三层用例链路（业务 → 系统 → 应用）

理由：DCDDP 是本工程对 [困难三·沟通带宽](three-challenges.md#三沟通带宽与丰度--io-问题) 的答案。需求 / 设计两段的信息密度最高、最容易因散文表达而流失精度——上游用结构化模型表达，下游（开发 / 测试）才能拿到能直接执行的指令。

涉及项目级模型变更时，必须在 §执行计划 步骤序列里加入"同步更新 `docs/modeling/`"——遵守"模型 / 代码一致性"硬约束（详见项目级 CLAUDE.md "## 系统建模方法论"段）。

规范来源（agent 按需 Read）：`<aie-root>/methodology/modeling-conventions.md`、`<aie-root>/methodology/meta-model.schema.yaml`、`<aie-root>/methodology/examples/chargable-proxy/`。

## PM 的极简分类（拉系统设计师之前）

PM 收到用户请求只分两类，不做更细判断：

- **纯粹改字 / 调样式 / 换变量名** → 直接拉开发，跳过设计师
- **其它一切** → 拉系统设计师

细判断是设计师的事。

## design.md 骨架

```markdown
# 设计：<title>

## 设计正文
（边界 / 接口 / 不变量 / 失败模式；不写代码）

## 待确认
- Q1: <问题>，选项 A/B
（无则省略本节）

## 执行计划

需要设计评审：yes/no — <理由>
需要测试用例先设计：yes/no — <理由>

步骤序列（PM 照办）：
1. <step>
2. <step>
...

## 卡住条件
（命中即 PM 停步回用户；按本任务难度自定，未自定义沿用下方默认）
```

**默认卡住条件**（设计师未自定义时 PM 沿用）：

- 设计评审连续 2 轮仍有 blocker
- 开发反馈实现成本 > 预期 2 倍
- 测试连续 2 轮失败且失败原因相同

## 两种暂停

| 类型 | 触发 | PM 姿态 | 期望用户回复 |
|------|------|---------|------------|
| **待确认** | 设计师 / 评审员在文档写 `Q:` | "问题：... 选项 A/B" | 短答复，立刻继续 |
| **卡住** | 命中 §卡住条件 | "搞不动了：<现象 + 已试过的>" | 介入调整方向 / 降级目标 / 放弃 |

PM 输出 schema 用不同字段承载——例如 `pending_confirmation: [...]` 和 `stuck: {reason, attempts}`——不混在一起。

## log.md 格式

```
[<ISO datetime>] <发起者> → <承接者>：<动作摘要>
[<ISO datetime>] <角色>：<产出摘要>（关键指标，如 blocker 数 / 测试通过失败数）
[<ISO datetime>] PM → 用户：【待确认】... / 【卡住】...
```

顺序追加。**不维护额外状态机**，PM 数次数靠 grep `log.md`——`log.md` 就是事实源。

## 目录结构

```
projects/<initiative>/tasks/<slug>/team/
  design.md
  log.md
  reviews/         # 设计评审产出
  test-cases/      # 测试用例设计产出
```

兼容现有 `projects/<initiative>/tasks/<slug>/` 路径。

## 与旧 aie-task-* 的关系

完全替代。旧模式的 4 个核心病灶在新模式里成了非目标：

| 旧模式痛点 | 新模式的反向设计 |
|-----------|----------------|
| 强制线性流水线 → 简单任务也走全流程，体感极慢 | 流程由设计师按本任务难度在 §执行计划里裁剪；PM 还有"纯粹改字直通开发"的极简分类 |
| ensemble × 2 → 同模型同 prompt，意见高度相关，徒增成本 | 评审员单跑一次，结构上独立即够 |
| 阶段关注点写成"该想什么"（描述性）→ 设计阶段陷入实现细节 | 反清单是 system prompt 第一段（禁令），越界拒答 |
| state.json / sentinel / backtrack_history → 主对话被状态读写淹没 | 只有 log.md 顺序追加，grep 即统计，零状态机 |

## 实现现状
