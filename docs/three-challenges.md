# 用好 AI LLM 的三个困难

> 探讨稿。和 `methodology.md` 平行：methodology 讲"怎么做"，本文尝试拆"为什么难"。

## 一、上下文管理 — 容量问题

**核心命题**：信息总量受限，重要信息容易被淹没。

- 关心的是"信息在不在 prompt 里、占多少 token、会不会被噪声盖过"
- 固定开销（系统指令、Skill 描述符、MCP 工具定义）一旦超标，留给动态对话的空间就被挤压
- 经验上，上下文用量较高时模型会出现退化迹象（具体阈值因任务而异）

**应对手段**：分层归属（CLAUDE.md / Skills / Rules / Hooks）、`/compact`、`/clear`、Subagents 隔离、HANDOFF.md。

### 本工程的具体答案：评估机制

分层只是原则，**还需要可重复的方式去评估各层是否健康**——尤其是 CLAUDE.md 这种半固定开销最容易膨胀。本工程通过两个 skill 实现：

- **`/aie-test-add <project>`** — 给受管工程添加测试用例（prompt + 判断标准），存到 `<target>/.claude/aie-tests/<name>.md`。测试用例是用户对"该工程的 AI 应当怎样回应"的预期固化。
- **`/aie-improve <project>`** — 评估受管工程的 AI 协作配置，输出改进建议。包含：
  - **静态体检**：CLAUDE.md 体量、逐条规则的三问评审（频率 / 性质 / 刚性）、重复与冲突检测、与现有 Skills/Hooks 的去重
  - **行为测试**：在 `<target>` 目录下用 `claude -p` headless 跑测试用例（真实加载该工程的 CLAUDE.md / Skills / Hooks），用 LLM-as-judge 按 criteria 判定
  - **汇总建议**：按 severity 排序，每条带 scope / issue / suggestion / rationale；默认只读，落地走 diff 确认

**三问框架**（评估每条 CLAUDE.md 规则时的判定准则）：

1. **频率**：每次对话都用得到？否 → 挪 `.claude/skills/` 按需加载
2. **性质**：是规则还是数据？数据 → 拆独立文件 + 留指针
3. **刚性**：靠 AI 理解够？还是必须强制？强制 → 升级为 Hook

通过这三问被淘汰的内容应该比留下的多。一个反直觉的点：**CLAUDE.md 越大，单条规则越没用**，因为注意力权重被整个文档摊薄——这是"上下文管理"和"关注点管理"两个困难的交叉处。

## 二、关注点管理 — 角色分工问题

**核心命题**：单个**全能 AI** 同时背太多 concerns 时产出质量下降；不同 concern 应交给不同**专职 AI**。

软件工程里"关注点分离"早就是常识——为什么开发、测试、评审不是同一个人做？因为同一个脑袋同时背太多目标时，每个目标都会打折，而且自评天然有偏。对 AI 来说同样成立，甚至更严重——AI 没有"换人"的成本，但也很容易因为方便而把所有事压给主对话。

容易出问题的场景：

- **自评偏差**：自己实现 + 自己 review，倾向于美化自己产出，挑不出深层问题
- **concern 互相挤占**：同时背"功能要跑通 + 代码要干净 + 测试要覆盖 + 文档要更新"，每个维度都打折
- **短期压过长期**：让代码跑起来的紧迫感会淹没"测试是否真的覆盖了关键路径"
- **协调者被淹没**：主对话同时做规划 + 执行 + 验收，规划判断的质量被执行细节挤压

**手段的本质**：拆 + 隔离 + 独立判断

- **拆**：把生命周期不同阶段（设计 / 实现 / review / test / docs）交给不同 skill / agent
- **隔离**：每个专职 AI 只看自己该看的上下文，避免被无关信号污染
- **独立判断**：关键决策（验收、review）由结构上独立的 agent 做，物理隔离自评循环

**张力**：拆得越细，协调成本越高——状态要外化、verdict 要合并、上下文要在专职 AI 之间传递。所以不是"越分越好"，而是"自评偏差严重的环节优先拆"。

### 本工程的具体答案：多维专职化

本工程在三个维度上做专职化拆分：

| 拆分维度 | 机制 | 解决的失效 | 在本工程的体现 |
|---------|------|-----------|----------------|
| 纵向（按阶段） | Task 工作流 9 个 skill | concern 互相挤占；主对话被执行细节淹没 | init / design / design-review / implement / code-review / test / docs / plan / next 各管一段，state.json 外化状态 |
| 横向（同阶段多视角） | Ensemble × 2 | 单点偏见、自评偏差 | design-review / code-review 同一产物跑 2 个独立 replica，verdict 合并 |
| 升维（验收剥离） | Hook-as-agent | 实现者评自己难挑刺 | TaskCompleted hook 拉起独立 agent，读 overview.md/tasks.md 评估目标是否真正达成，必要时 block |
| 调度（协调 vs 执行） | 主对话 + Subagent | 协调判断质量被执行上下文挤压 | 主对话只做编排，Explore / general-purpose / Plan 拿走具体调查与方案细节 |

**与困难一/三的边界**：CLAUDE.md 重注入、Skill 描述符常驻、UserPromptSubmit hook 看似也是"关注点机制"，但它们解决的是"通用约定有没有被想起来"——属于困难一（上下文分层）和困难三（带宽固化）。困难二专指**因为是同一个 AI 而产出降质**，解法核心是**换一个 AI 来做这件事**。

**`/aie-improve` 的位置**：它本身就是"评估配置 + 加测试用例 + 给建议"三件事拆给不同 subagent 做——评估角色不该同时设计测试，测试 judge 不该同时给改写建议。这是把困难二的方法论用在工具自身设计上。

## 三、沟通带宽与丰度 — I/O 问题

**核心命题**：人和 AI 之间的信息通道很窄，两端都有损耗。

**上行（人 → AI）**：

- 人脑里的意图丰富、模糊、带强先验和未明说的约束
- 键盘出口只有几十字/分钟，口述也快不到哪去
- 对不同的同事可以用不同的措辞和默契沟通；对 AI 只能用一种当下能想到的说法
- 截图、文件路径、错误日志、链接是**带宽放大器**——单条消息携带的信息密度远高于纯文字描述

**下行（AI → 人）**：

- 一次输出几百行 diff 或大段分析，人未必消化得动
- 看似详尽的总结可能掩盖了关键决策点
- AI 有责任主动精炼输出，不能把消化成本全甩给人

**固化机制**：Skills / Memory / CLAUDE.md / `/aie-apply` 本质都是"提前充值带宽"——把每次都要重说的东西沉淀下来，让单次对话不必从零开始。

### 本工程的具体答案：系统建模方法论（DCDDP）

固化机制（Skills/Memory/CLAUDE.md）解决的是"通用约定"的带宽问题，但**针对具体业务系统**的意图传递，带宽缺口更大——一段需求描述很难一次性把实体、关系、规则、用例、状态机说清楚。本工程的答案是把"业务/系统设计"本身从自然语言升级为**结构化模型**，作为人和 AI 共享的高保真载体。

**核心思路**（详见 system-modeling 规范，由 `/aie-apply` 按项目选择性启用）：

| 维度 | 朴素做法 | DCDDP 做法 | 带宽收益 |
|------|---------|-----------|---------|
| 表达媒介 | 自然语言 prompt | 4 视图 + 3 层用例的结构化模型文件 | 单位字符承载的语义量大幅提升 |
| 对话方式 | 命令式（"帮我加个 X 功能"） | 声明式（先在模型里表达 What 与约束） | AI 推导 How，不必每次重描述意图 |
| 持久性 | 对话窗口内 | 落盘到 `docs/modeling/`，跨会话复用 | 带宽一次性投入，多次摊销 |
| 双向性 | 用户描述 → AI 实现 | 用户读模型验证 AI 理解；AI 读模型获取意图 | 上下行同时受益 |
| 一致性强约束 | 易"先改代码下次补模型" | "模型/代码一致性"硬约束 override 任何阶段拆分建议 | 防止带宽载体退化为陈旧文档 |

**为什么这是带宽问题的解，不只是关注点问题的解**：硬约束部分确实是关注点管理（已记在困难二里），但**模型本身的存在**才是带宽放大器——没有模型时，再强的关注点机制也只能反复要求 AI "搞清楚需求"，而需求的丰度根本没有载体可承载。模型给意图一个**容器**，关注点机制只是保护这个容器不被绕过。

**和困难一的张力**：模型文件不进对话上下文，按需 Read。所以模型可以远大于 CLAUDE.md 能承载的体量，不会挤压上下文容量——这是为什么要把它放在文件系统而不是常驻 prompt。

**配套工具**：`/model-view` 启动 viewer 浏览模型；`/model-build` 出静态站。这些是带宽放大器在工程上的落地——让人能高效消化模型，反向减少 AI → 人的下行带宽压力。

## 三者的关系

| 困难 | 隐喻 | 主要工具 |
|------|------|---------|
| 上下文管理 | 仓库容量 | 分层归属（CLAUDE.md / Skills / Rules / Hooks）、压缩、隔离 |
| 关注点管理 | 分工 / 编制 | 专职 skill / subagent、ensemble × 2、hook-as-agent 验收 |
| 沟通带宽/丰度 | 信道宽度 | 通用固化（Skills/Memory）、业务建模（DCDDP）、放大器（截图/路径） |

**互相约束**：

- 拆专职 AI 越细 → 协调成本越高、状态外化越重、对带宽和上下文管理的要求越高
- 带宽载体（模型/CLAUDE.md）越丰富 → 单个 AI 越可能"自以为够了"不去拆专职——好的载体会掩盖角色分工的需要
- 上下文紧张 → 只能少拉 subagent、少跑 ensemble，被迫退回全能 AI

任何一个"解法"几乎都在三者之间做权衡，没有不付代价的方案。
