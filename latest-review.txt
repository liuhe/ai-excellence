  整体设计 Review

  P0（影响正确性 / 一致性）

  1. dispatcher 状态转移表残留无意义项
  - design-review 不可能 backtrack——它只有 pass/revise（上游就只有 design，revise 即等价
  backtrack(design)）。表里那行应删，避免 plan 错误产出 backtrack-to: design 让 dispatcher
  多绕逻辑。

  2. plan 的 exec_files 通过 prompt 注入是冗余
  - plan SKILL 完全可以从 stage + iteration 自己推：ensemble 看 <dir>/<S>-<NNN>a.md + b.md，单
  replica 看 <dir>/<S>-<NNN>.md。dispatcher prompt 里那段可简化掉，少一处可漂移的契约。

  3. sentinel 与 failure_log 字段不统一
  - plan sentinel 写 "stage": "plan-<S>"（复合字符串），failure_log 写 {stage: S, phase:
  "plan"}（结构化）。同一信息两种表示，未来如果有工具/脚本解析容易踩坑。统一为后者更好。

  4. code-review / plan 在 no-design 时找不到 design
  - 上面已说，加 fallback。

  P1（清晰度）

  5. 旧 state.json 迁移路径没写明
  - 已经初始化过 task 的 state.json 没有 phase 字段、consecutive_pass 含多键。dispatcher 该如何
  forward-compat：缺 phase → 默认 "exec"；多余 consecutive_pass 键 → 忽略。我在 init
  里说了，但没在 dispatcher 里说。

  6. "phase" 术语两层混用
  - state.phase（状态机内部字段）vs "plan phase / exec
  phase"（讨论用语）。文档里混着用。不影响代码正确性，读起来稍累。

  P2（架构 / 鲁棒性）

  7. ensemble 的"共因失败"风险
  - 两个 replica 用同 prompt、同 model、同 task_dir 上下文。如果两个 fresh-ctx agent
  都看不到某关键漏洞，plan 见"两人一致"会判 pass。原本 consecutive ×2 也有同问题（时间维度独立 ≠
  真独立），不算回归。可以未来引入"adversarial replica"（一个 b 用反向 prompt：刻意找 a
  看漏的）但属增量优化。

  8. plan worker 中途崩溃 → 副作用部分写入
  - docs plan 要写 progress.txt +
  docs-update-suggestions.md，如果中间崩溃，前者已写后者未写。没有事务。重试时 plan 看到
  progress.txt 有部分内容、suggestions.md 缺失——可能重复追加。
  - 化解：要么把多副作用写成"先 plan 文件落地 → 再串行追加" + 用 idempotent
  追加（先读再判断已写），要么接受一致性偶失（docs 阶段反正还要 ×2 收敛）。

  9. 同一 firing 内 exec→plan 串联的失败语义
  - exec ok 写 phase=plan 持久化后 spawn plan。如果 plan 这次 spawn 直接失败（Task tool
  异常等），主会话能不能 graceful 处理？SKILL 说"plan sentinel 缺失 → failure_log，保持
  phase=plan，下次 firing 重试"。但下次 firing 时主会话已经退出，得靠 ScheduleWakeup 续。我没在
  SKILL 里说"plan 这步失败时也要 ScheduleWakeup"——补一句更稳。

  10. 跨 stage 的"learning"丢失
  - 旧实现里 implement 写 progress.txt Codebase Patterns 是显式约定。新实现下 implement
  仍然写（无 plan phase 的 stage 自己写副作用），docs plan 也写。两个来源都往 Codebase Patterns
  段追加，没有协调机制。可能重复或冲突。
  - 化解：分工——implement 只写"实现过程踩坑"；docs plan 只写"docs 阶段发现的可复用模式"。在两处
  SKILL 里加注释明确分工边界。

  P3（小事）

  11. 没有 worker timeout：Task() spawn 没指定超时；hang 住会无限等。可能 Task tool 自身有默认
  timeout，但 SKILL 没记录。

  12. ensemble 命名 a/b 在文件名里：design-001a.md，目录里和 design-001-plan.md 混排，glob
  design-001* 会同时匹配三个。如果以后有解析脚本要小心。
