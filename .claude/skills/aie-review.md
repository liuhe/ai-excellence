# /aie-review [项目名]

审查项目的 AI 协作配置状态，基于 docs/methodology.md 方法论给出评分和建议。

## 行为

- 无参数：遍历 `projects/` 下所有软链工程，输出汇总报告
- 有参数：深入审查指定项目，给出详细建议

## 审查前提

**先理解项目在做什么，再判断分层。** 高频/低频的判断取决于项目定位——同样的内容在不同项目里性质不同。

## 审查核心原则

**最关注的问题是：设置是否放对了地方。**

按 docs/methodology.md 的分层思路，内容按性质归类：
- **核心概念和常驻约束** → CLAUDE.md（每次对话都需要的，基于项目定位判断）
- **参考数据和业务规则** → Rules 文件 + CLAUDE.md 指针（数据放独立文件，CLAUDE.md 只放一行指针告诉 Claude 去哪读）
- **低频操作流程** → Skills（按需触发的步骤序列）
- **硬性校验** → Hooks（不应只写在 CLAUDE.md 里当建议）
- **全局通用** → ~/.claude/（不应重复写在每个项目里）
- **项目特有** → 项目级配置（不应跑到全局去影响其他项目）

**次要关注**：配置是否存在、格式是否规范。缺少某项配置不一定是问题。

## 审查维度

### 1. 分层合理性（核心）
- [ ] CLAUDE.md 中是否有不属于核心概念的参考数据？（应放 Rules 文件）
- [ ] CLAUDE.md 中是否有低频操作流程？（应放 Skills）
- [ ] CLAUDE.md 中是否有应该用 Hooks 强制执行的规则？
- [ ] Skills 中是否有应该常驻在 CLAUDE.md 里的核心概念？
- [ ] 项目级设置中是否有应该放在全局的通用配置？
- [ ] 全局设置中是否有应该下沉到项目级的特有配置？

### 2. 现状记录（如实描述）
- CLAUDE.md：行数、内容概要
- settings.local.json：权限配置
- Skills：有哪些，做什么
- Hooks：有哪些，做什么
- MCP：有哪些
- README-AIE.md：是否存在

## 执行步骤

1. 读取 `docs/methodology.md` 作为评估标准
2. 确定要审查的项目范围
3. 对每个项目，逐层检查上述维度
4. 输出审查结果和发现的问题
5. 如果发现需要修复的问题，直接给出具体修复方案，询问用户是否执行
6. 用户确认后执行修复（如移动设置到正确位置、从模板生成缺失文件等）

## 输出格式

### 汇总模式（无参数）

对每个项目简要列出分层合理性问题和现状概要。

### 详细模式（指定项目）

逐层输出检查结果，对发现的问题直接提供修复方案供用户确认执行。
