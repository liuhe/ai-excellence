# DCDDP 愿景（v6.2）

> Domain & Component Driven Development Platform

## 一句话

**沉淀一套高效的系统建模方法论，让人和 AI 用同一种语言沟通系统逻辑。**

## 项目定位

DCDDP 是**方法论项目**，不是工具平台。它的产出不是"一个能跑的 IDE"，而是：

1. **一套建模方法论** — 约定、主要模型、组织方式、表达方式。
2. **一份可扩展的元模型 schema + 专用 viewer** — schema 定义结构，viewer 让结构变得可读、可关联、可跳转。
3. **一条应用集成路径** — 把方法论嵌入实际项目，让 AI 在那些项目里按这套方法论与用户沟通。

工具（schema/viewer）服务于方法论；样板（chargable-proxy）证明方法论能落地。

## 核心理念

### 声明式：用户定义 What，AI 推导 How

模型表达**意图与约束**，实现细节由 AI 填充。配置量应远小于等价代码量（< 10%）。

| 概念 | 用户定义（What） | AI 推导（How） |
|------|------------------|----------------|
| 领域模型 | 实体、属性、关系、状态机、规则 | DDL、索引、ORM 映射、迁移 |
| 用例 | 参与者、入口、规则、关联 | 接口实现、流程编排、权限拦截 |
| 系统拓扑 | 应用、技术栈约束、交互语义 | 部署、内部包结构、日志埋点 |

### 4 视图分层

模型按 4 个视图组织，回答不同读者的不同问题：

| 视图 | 回答 | 主要内容 |
|------|------|---------|
| **业务视图** | 谁在用、价值是什么、业务里有哪些核心概念 | 业务参与方、业务用例、系统对外能力（系统用例）、**业务模型**（业务实体，四色建模） |
| **应用视图** | 系统怎么实现 | 应用组成、应用拓扑、应用用例（含规则）、页面、**应用领域模型**（DDD 构造块，每个 app 各一份，可选） |
| **系统部署视图** | 系统部署在哪 | 节点、网络、配置 |
| ~~领域模型视图~~ | （已废止——业务实体下沉到业务视图，代码层结构变成各 app 的应用领域模型） | |

每个视图遵循 **Overview / Details 递归**：先一份精简概览，再按需展开 details，层数不限。

详见 `v6/v6.2/modeling-conventions.md` 与 `v6/v6.2/meta-model.schema.yaml`。

### 知识是关联的，所以 viewer 是必需品

模型不是孤立文件的集合：业务用例 → 系统用例 → 应用用例 → 页面 / 实体规则；实体 ↔ 应用用例规则双向追溯。这种关联只有专用 viewer 能高效呈现（点击跳转、跨文件导航、命名空间校验）。

普通 markdown 浏览器看不到这些关联，所以 viewer 不是"锦上添花"，而是方法论的必要组成。

### Schema 扩展点支持富文本

Schema 在合适的位置允许嵌入 markdown 描述（不止 URL 链接），用于补充：

- 业务背景、设计权衡、历史决策
- 内嵌图片 / SVG / 外部文章引用
- 不适合结构化的注意事项

URL 类字段保留（用于强关联跳转），markdown 字段补充（用于原位富文本说明）。两者并存。

## 应用方式

DCDDP 的落地路径是 **作为 `ai-excellence` 的扩展注入到受管工程**：

- 受管工程的开发者与该工程内的 AI 协作时，AI 已具备本方法论的知识。
- 讨论系统现状 / 变更前对齐 / 实现拆解，都按这套语言进行。
- 模型文件存放在受管工程内，由该工程的开发者维护。

`examples/chargable-proxy/` 样板是方法论的"参考实现"，给受管工程提供模仿对象。

## 设计原则

1. **AI 优先**：方法论首先要让 AI 能产出符合规范的模型；人是审阅者和补充者。
2. **小文件、强关联**：单文件可读，跨文件引用清晰，AI 处理友好。
3. **图不超载**：SVG 是 YAML 的可视化呈现，不能携带 YAML 之外的信息。
4. **路径灵活**：4 视图维度固定，但建模起点可变（需求驱动从业务用例 / 技术驱动从系统架构 / 数据中心型从领域模型）。
5. **扩展开放**：未来视图（如 security、events、workflows）作为独立顶级文件加入，不破坏既有结构。
