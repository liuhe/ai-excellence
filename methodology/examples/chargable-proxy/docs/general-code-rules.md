# 通用规则

本文件定义代码生成和测试时应遵循的通用规则和约定。

## 代码结构

### DDD 分层结构

各子系统应采用领域驱动设计（DDD）的分层结构：

- **Controller / Handler 层**：处理 HTTP 请求/响应，参数校验，不含业务逻辑
- **Service / Application 层**：编排业务逻辑，事务管理
- **Domain / Model 层**：领域模型和业务规则，不依赖基础设施
- **Repository / Infrastructure 层**：数据持久化，外部服务调用

要点：
- Domain Model 与持久化 Entity 分离，通过转换方法互转
- Service 层负责事务边界
- Controller 层不直接调用 Repository
- 对于关键的数据模型或用例规则，可抽出单独的策略类进行管理，提升可读性和可维护性

### 前端路由

前端应用使用 **基于 URL 的路由**，页面状态应反映在 URL 中，支持：
- 直接通过 URL 访问特定页面
- 浏览器前进/后退导航
- 页面参数通过 URL 路径或查询参数传递

## 可测试性

### 依赖抽象

- 外部依赖（数据库、HTTP API、文件系统等）应通过接口/抽象层隔离
- 提供内存实现（in-memory implementation）用于单元测试和本地开发
- 运行时通过配置选择具体实现

### 测试策略

- **单元测试**：使用内存实现替代外部依赖，测试业务逻辑
- **集成测试**：测试数据库访问、API 调用等基础设施层
- 关键业务规则必须有充分的单元测试覆盖

## 监控

### Prometheus 指标

- 各子系统应暴露 Prometheus 指标端点（`/metrics`）
- 至少包含：请求计数、错误率、关键业务指标（如流量统计）
- 使用有意义的标签（labels）区分维度

### 健康检查

- 各子系统应提供健康检查端点，供部署和运维使用

## API 规范

### 统一响应格式

面向用户的 API 使用统一的 JSON 响应包装：

```json
{
  "ok": true,
  "message": "ok",
  "data": <具体数据>
}
```

失败时：

```json
{
  "ok": false,
  "message": "错误信息",
  "data": null
}
```

内部 API 不使用此包装，直接返回数据或 HTTP 状态码。

## 并发安全

- 涉及状态变更的写操作（如兑换码绑定、套餐状态更新等）需保证并发安全，防止竞态条件导致数据不一致
- 优先使用数据库层面的原子操作（如条件 UPDATE）而非先查后改模式
- 对于需要跨多条记录的一致性操作，使用数据库事务保护

## 安全

- 管理员操作与普通用户操作应有独立的认证机制
- 敏感配置（数据库密码、密钥等）通过环境变量或配置文件注入，不硬编码

## 日志

- 使用结构化日志，包含时间戳和源文件位置
- 日志文件应配置轮转策略，避免磁盘空间耗尽
- 关键业务事件使用 INFO/WARN 级别记录
- 详细调试信息通过 verbose 开关控制

## 数据库

### DDL 文件

- 使用关系型数据库的子系统须在项目中提供独立的 DDL 文件（如 `schema.sql`），包含所有表的 CREATE TABLE 语句
- DDL 文件应可重复执行（使用 `IF NOT EXISTS` 等）
- 字段命名使用 snake_case，与代码中的领域模型通过 ORM 映射或手动转换对应
- 必须包含必要的索引定义（主键、唯一约束、查询高频字段的索引）
